访问数据库

对于数据的库的开发一直都是Java项目中的重点内容。在Java的发展历史中，数据库持久层的主流技术随着时代的变化也发生了巨大的变化。

Spring支持JdbcTemplate的数据访问模式，但这个用法一直未被很多的企业采用。不过正如Spring所倡导的理念，它并不排斥其他优秀的框架，而是通过提供各种各样的模板，使得这些框架能够整合到Spring中来，并且更加的方便开发者的使用。

1、配置数据源

在依赖于Spring Boot的spring-boot-starter-data-jpa后，它就会默认配置数据源，这些默认的数据源主要是内存数据库，如h2、hqldb和Derby等内存数据，有时候需要配置为我们想要的数据源。

配置默认的数据源

以h2数据库为例，先添加依赖：

        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-data-jpa</artifactId>
        </dependency>

        <dependency>
            <groupId>com.h2database</groupId>
            <artifactId>h2</artifactId>
            <scope>runtime</scope>
        </dependency>

因为h2是内嵌式数据库，当然我们也可以将数据源配置为hqldb或者Derby，它会随着Spring Boot项目启动而启动，并不需要任何配置，单着显然不是我们所期待的。

配置自定义数据源

下面以MySql作为自定义数据源。

添加依赖

<!-- https://mvnrepository.com/artifact/mysql/mysql-connector-java -->
        <dependency>
            <groupId>mysql</groupId>
            <artifactId>mysql-connector-java</artifactId>
            <version>8.0.18</version>
        </dependency>
        <!-- https://mvnrepository.com/artifact/org.springframework.boot/spring-boot-starter-jdbc -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-jdbc</artifactId>
            <version>2.2.6.RELEASE</version>
        </dependency>

配置数据源

在application.properties配置文件已达到配置数据源的效果。在默认的情况下，Spring Boot会使用其绑定的Tomcat数据源。

spring.datasource.url=jdbc:mysql://localhost:3306/mybatis?useSSL=false&serverTimezone=UTC
spring.datasource.username=root
spring.datasource.password=panhao
#spring.datasource.driver-class-name=com.mysql.cj.jdbc.Drive
#最大等待连接中的数量，设0表示没有限
spring.datasource.tomcat.max-idle=10
#最大连接活动
spring.datasource.tomcat.max-active=50
#最大等待毫秒数，单位为ms，超过时间会出错误信
spring.datasource.tomcat.max-wait=10000
#数据库连接池初始化连接
spring.datasource.tomcat.initial-size=5

spring-datasource.driver-class-name=com.mysql.cj.jdbc.Driver表示配置数据源的驱动类型。这里注释掉了驱动，但是还是可以连接数据源的，这是因为Spring Boot会尽可能的去判断数据源是什么类型的，然后根据其默认的情况去匹配驱动类。在它不能匹配的情况下，可以明确的配置它，这样就不会使用默认的驱动类了。

上面只是匹配Spring Boot绑定的Tomcat的数据源，大多数的时候都会使用第三方的数据源，此时添加Maven依赖：

<!-- https://mvnrepository.com/artifact/org.apache.commons/commons-dbcp2 -->
        <dependency>
            <groupId>org.apache.commons</groupId>
            <artifactId>commons-dbcp2</artifactId>
            <version>2.1.1</version>
        </dependency>

添加了上述的依赖后，修改application.properties里面的内容：

spring.datasource.url=jdbc:mysql://localhost:3306/mybatis?useSSL=false&serverTimezone=UTC
spring.datasource.username=root
spring.datasource.password=panhao
spring.datasource.driver-class-name=com.mysql.cj.jdbc.Drive
#指定数据库连接池的类
spring.datasource.type=org.apache.commons.dbcp2.BasicDataSource
#最大等待连接中的数量，设0为没有限制
spring.datasource.dbcp2.max-idle=10
#最大连接活动
spring.datasource.dbcp2.max-total=50
#最大等待毫秒数，单位为ms，超过时间会出错误的信
spring.datasource.dbcp2.max-wait-millis=10000
#数据库连接池初始化连接数
spring.datasource.dbcp2.initial-size=5

2、使用JdbcTemplate操作数据库

在配置好数据源后，Spring Boot通过其自动配置机制配置好了JdbcTemplate，JdbcTemplate模板是Spring框架提供的。在实际的工作中还是比较少使用的。在实际工作中用的最多的是Mybatis。

创建一个Service接口，定义一些方法。这样通过它的实现类就可以注入Spring Boot已经为我们配置好的JdbcTemplate，直接使用就可以了。

3、整合Mybatis框架

目前Java持久层最为主流的技术已经是Mybatis，它比JPA和Hibernate更为简单易用。

Mybatis的框架配置

MyBatis是一个基于SQLSessionFactory构建的框架。对于SqlSessionFactory而言，它的作用是生成SqlSession接口对象，这个接口对象是Mybatis操作的核心，而在MyBatis-Spring的结合中甚至可以擦除这个对象，使其在代码中消失，这样做的意义是重大的，因为SqlSession是一个功能性的代码，擦除之后，就剩下了业务代码，这样就可以使得代码更具可读性。

因为SqlSessionFactory的作用是单一的，只是为了创建核心接口SqlSession，所以在MyBatis应用的生命周期中理当只存在一个SqlSessionFactory对象，并且往往会使用单例模式。而构建SqlSessionFactory是通过配置类（Configuration）来完成的。因此对于mybatis-spring-boot-starter，它会给予我们再配置文件中（application.properties）进行Configuration配置的相关内容。

  • poperties（属性）：属性文件在实际应用中一般采用Spring进行配置，而不是Mybatis。
  • setting（设置）：它的配置将改变MyBatis的底层行为，可以配置映射规则，如自动映射和驼峰映射、执行器（类型）、缓存等内容，比较复杂。
  • typeAliases（类型别名）：因为使用类的全限定名比较少，所以MyBatis会对常用的类提供默认的别名，此外还允许我们通过typeAliases配置自定义的别名。
  • typeHandler（类型处理器）：这是MyBatis的重要配置之一，在Mybatis写入和读取数据库的过程中对于不同类型的数据进行自定义转换，在大部分的情况下我们不需要使用自定义的typeHandler，因为在MyBatis自身就已经定义了比较多的typeHandler，MyBatis会自动识别javaType和jdbcType，从而实现各种类型的转换。一般来说，typeHandler的使用集中在枚举类型上。
  • objectFactory（对象工厂）：这是一个在MyBatis生成返回的POJO时会调用的工厂类。一般我们使用MyBatis默认提供的对象工厂（DefaultFactory）就可以了，而不需要任何配置。
  • plugins（插件）：有时候也称之为拦截器，是MyBatis最强大也是最危险的组件，它通过动态代理和责任链模式来完成，可以修改MyBatis底层实现的功能。
  • enviroments（数据库环境）：可以配置数据库连接内容和事务，一般而言，这些交由Spring托管。
  • mapper（映射器）：是MyBatis最核心的组件，它提供了SQL和POJO映射关系，这是MyBatis开发的核心。

步骤三：添加映射的实体类

创建User实体类，实体类与数据库的表相对应：

package com.panhao.demo.pojo;

import org.apache.ibatis.type.Alias;

import com.panhao.demo.pojo.enums.SexEnum;

import lombok.Getter;
import lombok.Setter;
import lombok.ToString;

@Getter
@Setter
@ToString
//Mybatis指定别名
@Alias(value = "user")
public class User {

    private Integer id =null;
    private String userName =null;
//    性别枚举，这里需要使用typeHandler进行转换
    private SexEnum sex=null;
    private String note=null;
}
  • @Alias 注解，用于指定实体类的别名，此处指定的别名是user;

值得注意的是此处性别的枚举类型，需要使用typeHandler进行转换。

package com.panhao.demo.typeHandler;

import java.sql.CallableStatement;
import java.sql.PreparedStatement;
import java.sql.ResultSet;
import java.sql.SQLException;

import org.apache.ibatis.jdbc.Null;
import org.apache.ibatis.type.BaseTypeHandler;
import org.apache.ibatis.type.JdbcType;
import org.apache.ibatis.type.MappedJdbcTypes;
import org.apache.ibatis.type.MappedTypes;

import com.panhao.demo.pojo.enums.SexEnum;

@MappedJdbcTypes(JdbcType.INTEGER)
@MappedTypes(value = SexEnum.class)
public class SexTypeHandler extends BaseTypeHandler<SexEnum> {

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i, SexEnum parameter, JdbcType jdbcType)
            throws SQLException {
        // TODO Auto-generated method stub
        ps.setInt(i, parameter.getId());

    }

    @Override
    public SexEnum getNullableResult(ResultSet rs, String columnName) throws SQLException {
        // TODO Auto-generated method stub
        int sex =rs.getInt(columnName);
        if(sex!=1&&sex!=2) {
            return null;
        }
        return SexEnum.getEnumById(sex);
    }

    @Override
    public SexEnum getNullableResult(ResultSet rs, int columnIndex) throws SQLException {
        // TODO Auto-generated method stub
        int sex=rs.getInt(columnIndex);
        if(sex!=1&&sex!=2) {
            return null;
        }
        return SexEnum.getEnumById(sex);
    }

    @Override
    public SexEnum getNullableResult(CallableStatement cs, int columnIndex) throws SQLException {
        // TODO Auto-generated method stub
        int sex=cs.getInt(columnIndex);
        if(sex!=1&&sex!=2) {
            return null;
        }
        return SexEnum.getEnumById(sex);
    }


}
  • @MappedJdbcTypes 声明JdbcType为数据库类型，此处对应的转换类型就是Integer；@MappedTypes 声明JavaType的类型，此处为SexEnum，这样MyBatis即可据此对对应的数据类型进行转换。

在MyBatis中，对于TypeHandler的要求是，实现TypeHandler<T>接口，而它自身为了更加方便也通过抽象类BaseTypeHandler<T>实现了TyperHandler<T>接口，所以这里直接继承了BaseTypeHandler<T>就可以了。

步骤四：添加映射文件

MyBatis文件中映射文件的具体形式就是XML文件：

<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE mapper
PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
"http://mybatis.org/dtd/mybatis-3-mapper.dtd">

<!-- nameSpace表示命名空间 -->
<mapper namespace="com.panhao.demo.dao.MyBatisUserDao">
    <!-- 根据客户编号获取客户信息 -->
    <select id="getUser" parameterType="Integer"
        resultType="com.panhao.demo.pojo.User">
        <!-- #{id}相当占位符 -->
        select * from t_user where id = #{id}
    </select>
</mapper>
步骤五：添加映射文件对应的操作接口

必须添加一个Java的接口，然后提供给Spring操作：

package com.panhao.demo.dao;

import org.springframework.stereotype.Repository;

import com.panhao.demo.pojo.User;

@Repository
public interface MyBatisUserDao {

    public User getUser(Integer id);
}
  • @Repository 用于标注数据访问组件，即DAO组件。这个注解在加载MyBatis接口Bean的时候是十分有用的，而它的方法getUser和映射文件中定义的查询SQL的id是保持一致的，参数也是如此，这样就能够定义查询方法了。

步骤六：测试MyBatis

MyBatis社区与Spring整合的包中提供了两个类，它们是MapperFactoryBean、MapperScannerConfigure。

  1. MapperFactoryBean 是针对接口的一个配置；
  2. MapperScannerConfigure 则是扫描装配，也就是提供扫描装配MyBatis的接口到Spring Ioc容器中。

实际上，MyBatis还提供了注解@MapperScan 也就能够将MyBatis所需对应的接口扫描装配到Spring Ioc容器中。

使用MapperFactoryBean装配MyBatis接口

在Spring Boot的启动配置文件中加入：

    @Autowired
    SqlSessionFactory sqlSessionFactory=null;

    @Bean
    public MapperFactoryBean<MyBatisUserDao> initMybatisUserDao(){
        MapperFactoryBean<MyBatisUserDao> bean=new MapperFactoryBean<MyBatisUserDao>();
        bean.setMapperInterface(MyBatisUserDao.class);
        bean.setSqlSessionFactory(sqlSessionFactory);
        return bean;
    }

使用MapperScannerConfigurer装配MyBatis接口

    @Bean
    public MapperScannerConfigurer mapperScannerConfig() {
//        定义扫描器实例
        MapperScannerConfigurer mapperScannerConfigurer=new MapperScannerConfigurer();
//        加载SQLSessionFactory，Spring Boot会自动生产，SqlSessionFactory实例
        mapperScannerConfigurer.setSqlSessionFactoryBeanName("sqlSessionFactory");
//        定义扫描的包
        mapperScannerConfigurer .setBasePackage("com.panhao.demo.*");
//        限定被标注@Repository 的接口才能被扫描
        mapperScannerConfigurer.setAnnotationClass(Repository.class);
//        通过继承某个接口限制扫描，一般使用不多
//        mapperScannerConfigurer.setMarkerInterface(......);
        return mapperScannerConfigurer;
    }

使用@MapperScan装配MyBatis接口

//定义Spring Boot的扫描路径
@SpringBootApplication(ScanBasePackages={"com.panhao.demo"})
//定义JPA接口扫描包的路径
@EnableJpaRepositories(basePackages="com.panhao.demo.dao")
//定义实体Bean的扫描路径
@EntityScan(basePackages="com.panhao.demo.pojo")
//定义MyBatis的扫描
@MapperScan(
//指定扫描包
   basePackages="com.panhao.demo.*",
    //指定SqlSessionFactory，如果SqlSessionTemplate被指定，则作废
    sqlSessionFactoryRef="sqlSessionFactory",
    //指定SQLSessionTemplate，将忽略sqlSessionFactory配置
    sqlSessionTemplateRef="sqlSessionTemplate"
    //markerInterface=Class.class//限定扫描接口，不常用
    annotationClass=Repository.class
)
public static void main(String[] args) {
        SpringApplication.run(MyfirstSpringBootApplication.class, args);
    }

数据库事务处理

在Spring中，数据库事务是通过AOP技术来提供服务的。在JDBC中存在大量的try...catch...finally...语句，也同时存在大量的冗余代码，如那些打开和关闭数据库连接的代码以及事务回滚的代码。在使用Spring Aop之后，Spring将它们进行擦除了，可以看到更干净的代码。

对于一些业务网站而言，产品库存的扣减、交易记录以及账户都必须是要么同时成功，要么同时失败，这便是一种事务机制，对这样的机制数据库给予了支持。

1、JDBC的数据库事务

package com.panhao.demo.db.service.impl;

import java.sql.Connection;
import java.sql.PreparedStatement;
import java.sql.SQLException;

import javax.sql.DataSource;

import org.apache.ibatis.session.TransactionIsolationLevel;
import org.apache.ibatis.transaction.Transaction;
import org.springframework.beans.factory.annotation.Autowired;

import com.panhao.demo.db.service.JdbcService;

public class JdbcServiceImpl implements JdbcService {

    @Autowired
    private DataSource dataSource;

    public int insertUser(String userName, String note) {
        // TODO Auto-generated method stub
        Connection conn=null;
        int result=0;
        try {
            //获取连接
            conn=dataSource.getConnection();
//            开启事务
            conn.setAutoCommit(false);
//            设置隔离级别
            conn.setTransactionIsolation(TransactionIsolationLevel.READ_COMMITTED.getLevel());
//            执行SQL
            PreparedStatement ps=conn.prepareStatement("insert into t_user(user_name,note) values(?,?)");
            ps.setString(1, userName);
            ps.setString(2, note);
            result=ps.executeUpdate();
            //提交事务
            conn.commit();
        } catch (Exception e) {
//            回滚事务
            if(conn !=null) {
                try {
                    conn.rollback();
                } catch (SQLException e2) {
                    // TODO: handle exception
                    e2.printStackTrace();
                }
            }
            // TODO: handle exception
        }finally {
            try {
                if(conn!=null&&!conn.isClosed()) {
                    conn.close();
                }
            } catch (Exception e2) {
                // TODO: handle exception
                e2.printStackTrace();
            }
        }
        return result;
    }

}

以上就是完整的数据库工作流程，数据库连接的获取和关闭以及事务的提交和回滚，大量的try...catch...finally...语句。按照AOP的设计思想，就可以把除执行SQL这步之外的步骤抽取出来单独实现，这便是Spring数据库事务编程的思想。

2、Spring声明式事务管理

Spring声明式数据库事务约定

使用@Transactional 进行标注，告诉Spring在什么地方启用数据库事务功能。这个注解可以标注在类上或者方法上。

  1. 标注在类上时 ：代表这个类所有公共非静态的方法都将启用事务功能。
  2. 标注在方法上时 ：代表这个方法启用事务功能。

在@Transactional 中，还允许配置许多的属性，如事务的隔离级别和传播行为。当Spring的上下文开始调用被@Transactional标注的类或者方法的时候，Spring就会产生AOP的功能。

  • 如果没有发生异常，Spring数据库拦截器就会提交事务
  • 如果发生异常，就要判断一次事务定义器内的配置，如果事务定义器已经约定了该类型的异常不回滚事务就提交事务，如果没有任何配置或者不是配置不回滚事务的异常，则回回滚事务，并且将异常抛出，这步也是事务拦截器完成的。
  • 无论发生与否，Spring都会释放事务资源，这样就可以保证数据库连接池正常可用了。
package com.panhao.demo.db.service.impl;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.transaction.annotation.Transactional;

import com.panhao.demo.dao.UserDao;

public class UserServiceImpl implements UserService{

    @Autowired
    private UserDao userdao=null;

    @Override
    @Transactional

        // TODO Auto-generated method stub

    }

}

3、@Transactional的配置项

数据库事务属性都可以由@Transactional来配置。

@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Inherited
@Documented
public @interface Transactional {
    //通过bean name指定事务管理器
    @AliasFor("transactionManager")
    String value() default "";
    //同value属性
    @AliasFor("value")
    String transactionManager() default "";
    //指定传播行为
    Propagation propagation() default Propagation.REQUIRED;
    //指定隔离级别
    Isolation isolation() default Isolation.DEFAULT;
    //指定超时时间（单位秒）
    int timeout() default TransactionDefinition.TIMEOUT_DEFAULT;
    //是否只读事务
    boolean readOnly() default false;
    //方法在发生指定异常时回滚，默认是所有异常都回滚
    Class<? extends Throwable>[] rollbackFor() default {};
    //方法在发生指定异常名称时回滚，默认所有异常都回滚
    String[] rollbackForClassName() default {};
    //方法在发生指定异常时不回滚，默认是所有异常都回滚
    Class<? extends Throwable>[] noRollbackFor() default {};
    //方法在发生指定异常名陈时不回滚，默认所有异常都回滚
    String[] noRollbackForClassName() default {};
}

关于注解@Transactional 值得注意的是它可以放在接口上，也可以放在实现类上。但是推荐放在实现类上。如果使用在接口上，将不能使用CGLIB代理，只能使用JDK动态代理，并且使用对应的接口去代理该类，才能驱动这个注解，这将大大的限制使用。因此实现类上使用@Transactional注解才是最佳的方式。

4、Spring事务管理器

在Spring中事务管理器的顶层接口为PlatformTransactionManager，spring还为此定义了一系列的接口和类。引入了MyBatis以后最常用的事务管理器就是DataSourceTransactionManager，它也是实现了PlatformTransactionManager的类。

PlatformTransactionManager接口的源码：

package org.springframework.transaction;

import org.springframework.lang.Nullable;

public interface PlatformTransactionManager {
    //获取事务，它还会设置数据属性
    TransactionStatus getTransaction(@Nullable TransactionDefinition definition) throws TransactionException;
    //提交事务
    void commit(TransactionStatus status) throws TransactionException;
    //回滚事务
    void rollback(TransactionStatus status) throws TransactionException;

}
  • spring在管理事务的时候，就是将这些方法按照约定织入对应的流程中的，其中getTransaction方法的参数是一个事务定义器（TransactionDefinition），它是依赖配置的@Transactional的配置项生成的，于是通过它就能够设置事务属性了，而提交和回滚事务也就通过commit和rollback方法来执行。

6、隔离级别

现代的企业级应用对于高并发的例子十分的常见，对于同一件商品的库存，可能同时有多个线程在修改它的库存，时刻都是多个线程共享数据，这样就会在多线程的环境中扣减商品的库存。对于数据库而言，就会出现多个事务同时访问统一记录的情况，这样引起数据出现不一致的额情况，便是数据库的丢失更新问题。

数据库事务的知识

数据库事务具有以下四个基本特征（ACID）：

  1. 原子性（Atomic） ：事务中包含的操作被看作一个整体的业务单元，这个业务单元中的操作要么全部成功，要么全部失败，不会出现部分失败，部分成功的情况。
  2. 一致性（Consistency）：事务在完成时，必须使所有数据都保持一致的状态，在数据库中所有的修改都基于事务，保证了数据的完整性。
  3. 隔离性（Isloation）：为了压制数据库丢失更新的问题，数据库定义了隔离级别的定义，通过它的选择，可以在不同程度上压制丢失更新的发生。
  4. 持久性（Durability）：事务结束后，所有的数据都会固化到一个地方，如保存到磁盘中，即使断电重启也可以提供给应用程序访问。

丢失更新

  • 对于一个事务回滚另一个事务提交而引发数据不一致的情况，称之为第一类丢失更新。（目前大部分的数据库都克服了第一类丢失更新的问题，也就是现今数据库系统不会再出现第一类丢失更新的情况了）
  • 多个事务都提交引发的丢失更新称之为第二类丢失更新。

隔离级别

为了压制第二类丢失更新，数据库标准提出了4类隔离级别，在不同程度上压制丢失更新，这4类隔离级别是：未提交读、读写提交、可重复读和串行化。

  1. 未提交读（red uncommitted）是最低隔离界别，其含义是允许一个事务读取另外一个事务没有提交数据。未提交读是一种危险的隔离级别。

    优点 ：在于并发能力高，适合那些对数据一致性没有要求而追求高并发的场景。

    缺点 ：是出现脏读。

  2. 读写提交（read commited）隔离级别，是指一个事务只能读取另外一个事务已经提交了的数据，不能读取未提交的数据。

    优点 ：能很好的解决脏读的问题出现。

    缺点 ：会出现不可重复读的现象。

  3. 可重复读隔离级别的目标是克服读写提交中出现不可重复读的现象，因为在读写提交的时候，可能出现一些值的变化，影响当时事务执行。

    优点 ：克服不可重复读的现象。

    缺点 ：出现幻读的现象。

  4. 串行化（Serializable）是隔离的最高级别，它会要求所有的SQL都会按照顺序执行，这样就可以克服上述的隔离级别的出现的各种问题，所以它能够保证数据的一致性。

选择合理的隔离级别

隔离级别和可能发生的现象：

项目类型 脏读 不可重复度 幻读
未提交读 true true true
读写提交 false true true
可重复读 false false true
串行化 false false false

追求更高的隔离级别，它能更好的保证数据的一致性，但是也要付出锁的代价。有了锁，就意味着性能的丢失，而且隔离界别越高，性能越是直线下地下降。

7、传播行为

传播行为是方法之间调用事务采取的策略问题。在绝大部分的情况下，数据库的事务是要么都成功要么都失败。假如有一大笔的交易记录，绝大部分的交易都是可以完成，但是有一小部分因为特殊原因导致失败，这是不应该因为这一小部分的交易，而回滚整个交易。最理想的做法是，调用多个交易的时候，如果有一些交易发生异常，只是回滚那些出现异常的交易 。

在Spring中，当一个方法调用另外一个方法的时，可以让事务采取不同的策略工作，如新建事务或者挂起当前事务等，这便是事务的传播行为。简单来说，就是执行一个批量任务的批量事务的时候，当它调用单个交易时，让每一个交易不在当前事务中执行，而是创建一个新事务去执行该交易。

传播行为的定义

在spring事务机制中对数据库存在7种传播行为，它是通过枚举类型Propagation定义的。

public enum Propagation {

    /**
    需要事务，它是默认传播行为，如果当前存在事务，就沿用当前事务，否则新建一个事务运行子方法
    **/
    REQUIRED(TransactionDefinition.PROPAGATION_REQUIRED),
    //支持事务，如果当前存在事务，就沿用当前事务，如果不存在，则继续采用无事务的方式运行子方法
    SUPPORTS(TransactionDefinition.PROPAGATION_SUPPORTS),
    //必须使用事务，如果当前没有事务，则会抛出异常；如果存在当前事务，就沿用当前事务
    MANDATORY(TransactionDefinition.PROPAGATION_MANDATORY),
    //无论当前事务存不存在，都会创建新的事务运行方法，这样新事务就可以拥有新的锁和隔离级别的特性，与当前事务相互独立
    REQUIRES_NEW(TransactionDefinition.PROPAGATION_REQUIRES_NEW),
    //不支持事务，当前存在事务，将挂起事务，运行方法
    NOT_SUPPORTED(TransactionDefinition.PROPAGATION_NOT_SUPPORTED),
    //不支持事务，如果当前方法存在事务，则抛出异常，否则继续使用无事务机制运行
    NEVER(TransactionDefinition.PROPAGATION_NEVER),
    //在当前方法调用子方法的时，如果子方法发生异常，只回滚子方法执行过的SQL，而不回滚当前的事务
    NESTED(TransactionDefinition.PROPAGATION_NESTED);


    private final int value;


    Propagation(int value) {
        this.value = value;
    }

    public int value() {
        return this.value;
    }

}

最常用的回滚事务有三种：REQUIRED(TransactionDefinition.PROPAGATION_REQUIRED)、REQUIRES_NEW(TransactionDefinition.PROPAGATION_REQUIRES_NEW)、 NESTED(TransactionDefinition.PROPAGATION_NESTED)

传播行为的设置

@Transactional(isloation=Isolation.READ_Commited,propogation=Propagation.REQUIRED)
public int insertUser(List<User>,userList){
    int count=0;
    for(User user:userList){
        //调用方法，将使用@Transactional定义的传播行为
        count +=userService.insertUser(user);
    }
    return count;
}
